## Purpose

This document summarizes the safety guardrails for the `nanobk cf dns owner-smoke-create` harness.
This harness is for owner-approved disposable smoke testing only.
It must never be used for production proxy/web DNS creation.

## Core Rules

1. **Never use this harness for proxy/web production creation.**
   This is a disposable smoke test tool, not a DNS management tool.

2. **Only disposable `nanobk-smoke-*` labels.**
   Labels must start with `nanobk-smoke-` and contain only alphanumeric characters and hyphens.
   Production labels (`proxy`, `web`, `www`, `@`, `*`, `api`, `cdn`, `mail`, etc.) are blocked.

3. **Always owner approved.**
   `--owner-approve` must be explicitly provided.

4. **Always exact confirmation.**
   `--confirm-disposable-smoke` must match the exact phrase:
   `I UNDERSTAND THIS WILL CREATE AND DELETE ONE DISPOSABLE CLOUDFLARE DNS RECORD`

5. **Always cleanup.**
   `--cleanup` must be explicitly provided.
   No persistent DNS changes should remain after a run.

6. **Always post-check.**
   After create, a GET must confirm the record exists with matching name, type, and content.

7. **Always leftover check.**
   After cleanup, a GET must confirm no matching target record remains.

8. **Never overwrite existing records.**
   If a record with the target name/type already exists, the harness blocks and does not POST.

9. **Never force.**
   `--force`, `--overwrite`, `--apply`, `--yes`, `--keep-for-debug` are all rejected.

10. **Content validation requires public routable A/AAAA.**
    Private, link-local, unspecified, loopback, multicast, and reserved addresses are rejected.
    Documentation ranges (e.g., `203.0.113.0/24`, `2001:db8::/32`) are rejected.

11. **Post-check requires name/type/content match.**
    The post-check must find a record matching the exact name, type, and content created.

12. **Cleanup verification requires no matching target record remains.**
    Unrelated records with different content do not block cleanup verification.

13. **Never print secrets or IDs.**
    Token, credential path, zone ID, record ID, raw API URL, and raw API response must never appear in text, JSON, error, or debug output.

14. **If cleanup fails, stop and require manual dashboard verification.**
    The harness prints a safe warning without exposing secrets.

15. **Scanner false-positive handling.**
    If a scanner flags `raw_api_response_printed=false` as a raw API response leak, treat it as a possible false positive.
    Manually verify that no actual API URL, path, body, record ID, zone ID, token, or credential path was printed.
    The field `raw_api_response_printed` is a safe boolean status indicator, not a raw API response.

## Forbidden Actions

- POST for proxy/web production records
- PATCH/PUT on any record
- DELETE of pre-existing records
- Force or overwrite
- Printing token, credential path, zone ID, record ID, raw API URL, or raw API response
- Bot/Web/installer integration
- Full Wizard auto-apply
- Release/tag without explicit approval
