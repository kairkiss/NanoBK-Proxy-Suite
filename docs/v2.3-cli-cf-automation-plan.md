# v2.3 CLI Cloudflare/DNS/Cert/Token Automation Plan

## 1. Current Repo Inventory

### CLI Entry

- `bin/nanobk` — main CLI, subcommands: `home`, `setup status`, `setup wizard`, `setup dns`, `setup profile`, `cf dns`, `cf zones`, `vps ip`

### Installer Behavior

- `installer/bootstrap.sh` — clone/update repo, prepare CLI
- `installer/install.sh` — interactive VPS deploy, Cloudflare, Bot, Web; cert mode: self-signed or existing
- `installer/install-vps.sh` — Hysteria2/Reality proxy install
- `installer/install-cloudflare.sh` — Worker deploy, token generation (SUB_TOKEN, ADMIN_TOKEN, NANOB_TOKEN)

### Cloudflare Zone Discovery

- `lib/nanobk_cf_zones.py` — GET-only zone list via Cloudflare API

### VPS IP Detection

- `lib/nanobk_ip_detect.py` — dry-run public IPv4/IPv6 detection

### DNS Availability

- `lib/nanobk_cf_dns_availability.py` — GET-only subdomain record check

### DNS Plan/Preflight

- `lib/nanobk_cf_dns_plan_generator.py` — plan-only: IP + availability -> plan summary
- `lib/nanobk_cf_dns_create_preflight.py` — plan-only: preflight wrapper

### Owner Smoke Create

- `lib/nanobk_cf_dns_owner_smoke_create.py` — controlled disposable create + cleanup, owner-approved only

### Worker/Subscription Token Current State

- Tokens generated in `install-cloudflare.sh` via `generate_token()` (openssl rand)
- `bin/nanobk` reads token presence/fingerprint for home display, never prints raw token
- No CLI command to rotate tokens

### Certificate Current State

- `install.sh` offers: self-signed, existing cert, letsencrypt (placeholder)
- No standalone cert automation CLI command
- No ACME/Let's Encrypt integration in lib/

### Setup Profile/Home Current State

- `lib/nanobk_setup_profile.py` — save/show/clear local profile (zone, api-env, nodes)
- `lib/nanobk_setup_status.py` — unified setup status / CLI home
- `lib/nanobk_setup_wizard.py` — guided interactive wizard
- `lib/nanobk_dns_setup_assistant.py` — beginner DNS setup assistant

### Web/Bot Current State

- Web: `GET /api/home` read-only, requires login, calls `nanobk home --json`, redacts output
- Bot: `/home`, `/setup_status` owner-only, calls adapter, redacts output
- Both are read-only bridges, no live DNS/CF apply

## 2. v2.3 Product Goal

- User types `nanobk` — gets a clean, branded CLI console shell
- Beginner-friendly Cloudflare/DNS/cert/token setup flow
- Hide engineering logs by default, show simple language
- Explain "what's next" after every step
- Full flow: account -> zone -> domain -> DNS -> cert -> token -> ready

### v2.3.4 DNS Apply Engine with Confirmation

- plan -> explain -> confirm -> apply -> verify

### v2.3.5 Certificate Automation

- ACME/Let's Encrypt integration
- Auto-issue cert for proxy domain
- Replace self-signed default
- Cert renewal awareness
- **Status**: Preflight implemented (`nanobk setup cert plan`).

### v2.3.6 Certificate Issue Gate

- Gated cert issue: `nanobk setup cert issue --issue --confirm "..."` 
- Plan-only default; real issue requires exact confirmation
- acme.sh DNS-01 integration (no installcert, no reloadcmd)
- **Status**: Issue gate implemented.

### v2.3.7 Subscription Token Rotation

- CLI command to rotate SUB_TOKEN/ADMIN_TOKEN
- Update Worker with new token
- Verify subscription still works
- Backup old token before rotate
- **Status**: Rotation gate implemented (`nanobk setup token rotate`). Production Worker blocked by default.

### v2.3.8 Full CLI Setup Flow Integration

- Single `nanobk setup flow` / `nanobk setup run` command runs full read-only flow
- Zone -> DNS -> cert -> token -> verify
- Each step shows progress and next action
- Dangerous actions only shown as copy-paste commands with exact confirmation
- **Status**: Setup flow implemented. Read-only orchestration across all stages.

### v2.3.9 Real VPS Acceptance

- End-to-end test on real VPS
- Verify proxy works after full setup
- Verify subscription URL works
- Verify cert is valid
- **Status**: Real VPS acceptance gate implemented. All v2.3.0-v2.3.8 regression tests pass. Next stage should be explicit owner-approved production rollout, not automatic.

## 5. Safety Rules

- DNS apply must follow: plan -> explain -> confirm -> apply -> verify
- Never overwrite existing non-NanoBK DNS records
- Never delete records not created by NanoBK
- Never print token, zone ID, record ID, raw API response, private key, subscription URL
- Web/Bot remain read-only, no live DNS apply bridge
- No DNS mutation in v2.3.0 scope
- Production DNS create blocked until v2.3.4 at minimum
- Certificate automation preflight done in v2.3.5; issue gate still blocked
- Token rotation blocked until v2.3.6

## 6. Real Acceptance Philosophy

- v2.3 prioritizes real VPS acceptance over form tests
- Mock tests only for critical safety boundaries (redaction, no-mutation)
- "Can run, user can understand, recoverable" beats "formally perfect"
- Each version must pass on a real VPS before closeout
- Smoke tests run against real Cloudflare API when owner-approved
