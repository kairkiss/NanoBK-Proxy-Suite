# NanoBK Proxy Suite — v2.1.13 Controlled DNS Profile Generator Design Spec

## 1. Purpose

v2.1.4–v2.1.12 completed the read-only DNS preparation foundation. v2.1.13 is a
design-only bridge toward controlled local DNS profile generation.

The future generator may eventually write a DNS profile file, but this spec itself
does not implement writing. The generator must not perform DNS apply or Cloudflare
mutation.

## 3. Existing DNS Profile Schema

The current schema consumed by existing DNS helpers:

**Required fields:**
- `zoneName` — DNS zone/domain name
- `nodePrefix` — single DNS label, must not contain dots
- At least one of `ipv4` or `ipv6`

**Optional fields:**
- `zoneId` — may exist but Cloudflare apply/check still requires api-env zone binding
- `ipv4` — top-level string field, IPv4 address
- `ipv6` — top-level string field, IPv6 address
- `defaultProxied` — must be `false` if present
- `reserved.panelPrefix` — DNS label for future Cloudflare Tunnel
- `reserved.nanokPrefix` — DNS label for future Worker custom domain
- `reserved.nanobPrefix` — DNS label for future Worker custom domain

**Schema rules:**
- IPv4 is a top-level string field named `ipv4`.
- IPv6 is a top-level string field named `ipv6`.
- `nodePrefix` is a single DNS label and must not contain dots.
- `zoneName` is a DNS zone/domain name.
- `zoneId` may exist in the profile but Cloudflare apply/check still requires api-env zone binding.
- `defaultProxied` must be `false` if present.
- DNS planner generates DNS-only / `proxied=false` records.
- Ownership marker is generated by apply path, not stored in the profile.
- Unknown fields and secret-like fields must be rejected.
- Existing `validate_profile()` and planning logic should be reused instead of duplicating schema rules.

## 4. Current Full Wizard Behavior

Current Full Wizard DNS behavior (not changed by v2.1.13):

- Full Wizard currently collects zone/domain, node prefix, IPv4, optional IPv6.
- It writes `/etc/nanobk/cloudflare-dns-profile.json`, or a test temp path under `NANOBK_TEST_TMPDIR`.
- It writes via temp file + `chmod 600` + rename.
- It runs `validate-profile`.
- It runs local plan.
- It may ask before GET-only `apply --check`.
- It never auto-runs `apply --yes`.
- It prints manual apply guidance only.

v2.1.13 does not modify Full Wizard. Future generator must remain separate until Full Wizard integration is separately planned.

## 5. Key Design Problem

The v2.1 read-only DNS report intentionally contains redacted IPs and redacted
hostnames. Therefore, the report proves readiness but cannot be used as raw input
to generate a real DNS profile.

Future generator input must come from explicit raw user-provided values or another
explicitly scoped raw-value source.

**Do not reconstruct raw profile values from redacted report output.**

## 6. Recommended Command Roadmap

### Phase 1 — Preview-only (no file write)

```
nanobk cf dns profile preview --zone DOMAIN --node NODE --ipv4 VALUE [--ipv6 VALUE] [--json]
```

Rules:
- No file write.
- No Cloudflare call.
- No DNS apply/check.
- No raw IP output by default.
- Validates proposed profile locally.
- Returns masked preview and validation status.

### Phase 2 — Temp output writer

```
nanobk cf dns profile generate --zone DOMAIN --node NODE --ipv4 VALUE [--ipv6 VALUE] --output PATH --yes [--json]
```

Rules:
- Temp output path only.
- No production `/etc` path yet.
- Writes atomically.
- `chmod 600`.
- Refuses overwrite.
- Validates generated profile.
- No Cloudflare call.
- No DNS apply/check.

### Phase 3 — Production path writer

```
nanobk cf dns profile generate --zone DOMAIN --node NODE --ipv4 VALUE [--ipv6 VALUE] --output /etc/nanobk/cloudflare-dns-profile.json --confirm-hostname HOSTNAME [--replace]
```

Rules:
- Separate later stage only.
- Requires typed confirmation.
- Refuses overwrite by default.
- Replacement requires backup.
- `chmod 600`.
- No DNS apply/check.
- No Cloudflare mutation.

## 7. Input Model

- `zone` is required.
- `node` is required.
- `ipv4` is required unless IPv6-only is explicitly approved later.
- `ipv6` is optional.
- `api-env` is not required for pure profile generation.
- Raw IP args may be accepted in future implementation, but must never be echoed in stdout/stderr/JSON by default.
- No live IP detection.
- No external IP echo.
- No real interface reads.
- No Cloudflare call.
- Read-only report JSON is readiness evidence only, not raw profile input.
- IPv4-only and dual-stack should be supported.
- IPv6-only should be deferred or explicitly risk-reviewed.
- Production mode should require public/global IPs.
- Tests may use documentation IP ranges only under explicit test/fixture mode.

## 8. Output and Redaction Model

- Text output shows masked IPs and redacted hostnames only.
- JSON output contains no raw IPs by default.
- Generated profile file necessarily contains raw IP values, but the CLI must say raw values were stored and not printed.
- Raw zone/domain should be redacted in JSON output where possible.
- No token/env values are involved or printed.
- No protocol links or subscription URLs are printed.

## 9. Local File Write Model

For future writer stages:

- Default mode is no write.
- `--output PATH` required for write stage.
- Production `/etc` path is deferred to later stage.
- Never overwrite existing file by default.
- Replacement requires backup and typed confirmation.
- Write temp file in same directory.
- `chmod 600` before rename.
- Create parent directory with restrictive permissions where needed.
- Validate JSON before final write.
- Validate profile schema before final write.
- No partial file on failure.
- Clear recovery instructions.

## 10. Confirmation Model

- Preview needs no confirmation because it writes nothing.
- Temp output writer may use `--yes`.
- Production path writer requires typed confirmation.
- Existing file replacement requires backup and explicit `--replace` flow.
- Typed confirmation should use exact hostname internally but should not print raw hostname back by default.
- No prompt should suggest DNS has been applied.

## 11. Status Model

Future JSON fields:

| Field | Values | Meaning |
|-------|--------|---------|
| `ok` | `true`/`false` | Command/report construction result |
| `dns_mutation` | `false` always | No DNS records created/updated/deleted |
| `cloudflare_mutation` | `false` always | No Cloudflare API mutation |
| `dns_apply` | `false` always | No `cf dns apply` invoked |
| `profile_written` | `true`/`false` | Whether profile file was written |
| `profile_write_mode` | `preview`/`dry_run`/`temp_output`/`production` | Write mode |
| `output_path_class` | `none`/`temp`/`production` | Output path classification |
| `backup_created` | `true`/`false` | Whether backup was created before replacement |
| `validation_status` | `passed`/`failed`/`skipped` | Profile validation result |
| `next_step` | string | Guidance for next action |

Local profile writing is a filesystem mutation, separate from DNS mutation. Do not rely only on `mutation=false`. Always distinguish:
- `profile_written`
- `dns_mutation`
- `cloudflare_mutation`
- `dns_apply`

## 12. Validation Rules

Future generator validation:

- Reuse `validate_profile()`.
- Reuse planning logic where safe.
- Reject invalid zone.
- Reject invalid node.
- Reject invalid IP syntax.
- Reject loopback/multicast/reserved/private production IPs.
- Allow documentation IPs only in explicit test/fixture mode.
- Require at least one IP.
- `defaultProxied` must be `false`.
- Unknown fields rejected.
- Secret-like fields rejected.
- No apply/check validation in generator.

## 13. Test Strategy

### Preview tests:
- Valid IPv4-only.
- Valid dual-stack.
- Invalid zone.
- Invalid node.
- Invalid IPv4.
- Invalid IPv6.
- Private IP rejected in production mode.
- Documentation IP allowed only in test mode.
- No raw IP in stdout/stderr/JSON.
- No DNS apply/check.
- No Cloudflare calls.

### Temp writer tests:
- Writes JSON to temp output.
- `chmod 600`.
- Schema valid.
- `validate-profile` passes.
- Dry-run writes nothing.
- Existing file refuses overwrite.
- No partial file on failure.
- No raw IP output.
- No Cloudflare calls.
- No apply/check.

### Production writer tests (later only):
- Production path requires typed confirmation.
- Existing production profile refuses overwrite.
- Backup created before replacement.
- Backup `chmod 600`.
- Rollback/recovery message.
- No DNS apply/check.

### Source checks:
- No `cf dns apply`.
- No `apply --check`.
- No POST/PATCH/DELETE/PUT.
- No curl/wget.
- No external IP echo.
- No interface reads.
- No raw env read/source/eval.

## 14. Risk Assessment

| Risk | Mitigation |
|------|------------|
| Wrong profile written | Validate before write; preview first |
| Existing profile overwritten | Refuse overwrite by default; backup required for replacement |
| Raw IP/domain leaked in output | Mask IPs/domains in all CLI output |
| User confuses local profile write with DNS apply | Clear messaging; `dns_apply=false` always |
| Convenience path accidentally runs apply/check | No apply/check in generator |
| Full Wizard behavior widened accidentally | Generator remains separate until integration planned |
| Tests touch `/etc` | Test mode uses temp paths only |
| Private IP accepted unintentionally | Production mode rejects private IPs |
| Redacted report data used incorrectly as raw input | Document that report is readiness evidence only, not raw input |
