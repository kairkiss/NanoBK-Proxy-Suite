# v2.6.5 Legacy VPS Install Inventory

This inventory grounds the v2.6.5 product wrapper in the existing stable VPS
installation assets. v2.6.5 does not rewrite the legacy installer and does not
change its stable behavior.

## Legacy Installer Entrypoints

- User-facing legacy entrypoint: `bin/nanobk install`
- Unified installer: `installer/install.sh`
- Stable VPS four-protocol installer: `installer/install-vps.sh`

The current `bin/nanobk install` path invokes the unified installer with
`installer/install.sh --repo-dir "$REPO_DIR"`. The unified installer then builds
the VPS deployment command around `installer/install-vps.sh --yes ...` for the
VPS service installation step.

## Four-Protocol Assets

The stable VPS installer installs and configures these four proxy channels:

- `hy2` / Hysteria2
- `tuic` / TUIC v5
- `reality` / VLESS Reality
- `trojan` / Trojan TLS

Profile and secret generation is implemented in:

- `vps/lib/profile.sh`

Runtime profile and secret files are:

- `/etc/nanobk/profile.current.json`
- `/etc/nanobk/profile.initial.json`
- `/etc/nanobk/secrets.private.env`
- `/etc/nanobk/config.env`

The profile schema contains the protocol sections:

- `hy2`
- `tuic`
- `reality`
- `trojan`

Configuration templates are:

- `vps/templates/hysteria2.config.yaml.tpl`
- `vps/templates/tuic-v5.config.json.tpl`
- `vps/templates/xray-reality.config.json.tpl`
- `vps/templates/xray-trojan.config.json.tpl`

Production config destinations used by the legacy installer are:

- `/etc/hysteria/config.yaml`
- `/etc/proxy-stack/tuic-v5-9443/config.json`
- `/etc/proxy-stack/xray-reality-8443/config.json`
- `/etc/proxy-stack/xray-trojan-2443/config.json`

## Services Controlled By Legacy Installer

Systemd unit templates are:

- `vps/systemd/hysteria-server.service.tpl`
- `vps/systemd/tuic-v5-9443.service.tpl`
- `vps/systemd/xray-reality-8443.service.tpl`
- `vps/systemd/xray-trojan-2443.service.tpl`

installer are:

- `hysteria-server.service`
- `tuic-v5-9443.service`
- `xray-reality-8443.service`
- `xray-trojan-2443.service`

The legacy installer enables and starts these services with systemd after

## Healthcheck

Repository healthcheck source:

- `vps/scripts/healthcheck.sh`

Installed healthcheck path:

- `/opt/nanobk/bin/healthcheck.sh`

The healthcheck is read-only and checks services, ports, configs, and the local
profile. v2.6.5 does not run it automatically in the default wrapper path.

## Cloudflare And Worker Coupling Points

The VPS install generates the profile consumed by Cloudflare Worker flows. The
existing status logic also checks these local Cloudflare/Worker files:

- `.cloudflare.local.env`
- `.nanob.local.env`
- `/root/.nanok-cf-admin.env`

Existing Worker and subscription coupling points include:

- `installer/install-cloudflare.sh`
- `workers/nanok-worker.js`
- `workers/nanob-worker.js`
- `SUB_TOKEN`
- `ADMIN_TOKEN`
- `NANOK_*`
- `NANOB_*`

v2.6.5 does not deploy Worker, upload profile data, rotate tokens, or print any
Cloudflare/Worker secret values.

## New Wrapper Calls

The v2.6.5 wrapper is:

- `lib/nanobk_production_vps_install.py`

It calls or reads only these local, safe interfaces in the default path:

- `nanobk_domain_selection.load_selected_domain()` to read the selected domain.
- `nanobk_setup_profile.load_profile()` as a fallback for the selected domain.
- `/etc/nanobk/profile.current.json` for protocol-section presence only.
- `systemctl is-active <service>` for read-only service state when available.

It does not call the stable VPS installer in the default path.

The guarded real adapter is intentionally not connected in v2.6.5. If connected
later, it must call the discovered stable VPS installer entrypoint only:

- `installer/install-vps.sh`

That future call must remain behind both:

- exact confirmation phrase
- `NANOBK_ALLOW_REAL_VPS_INSTALL=1`

and must redact installer output before presenting anything to the user.

## Safety Risks

The legacy VPS installer is a real system installer. Risks that remain gated:

- starting, enabling, restarting, or reloading services
- overwriting existing protocol configs
- writing new local credentials
- exposing `/etc/nanobk/secrets.private.env`
- exposing private keys, UUIDs, passwords, or profile contents
- partial installs with some services/configs present
- token or protocol key rotation
- Worker profile upload or route deployment
- certificate issuing
- DNS or Cloudflare mutation

The v2.6.5 wrapper detects complete and partial installs before execution. A
complete existing install is treated as ready. A partial install blocks with
`next_step=repair_or_review`. No real install is performed by default.
